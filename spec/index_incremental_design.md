# Index 增量更新设计说明

## 1. 设计目标

`index` 命令的核心性能目标是：对于未发生变化的笔记，跳过文件 IO 和内容解析，只处理真正发生变更的文件。同时保证 backlinks 等全局反向索引的正确性。

---

## 2. 推荐方案：三阶段增量更新

### 阶段一：变更检测

扫描文件系统，对每个 Markdown 文件读取 `path`、`mtime`、`size`，与数据库现有记录对比，将文件分为三类：

- **新增**：数据库中不存在该 path
- **修改**：`mtime` 或 `size` 与数据库记录不一致
- **删除**：数据库中存在该 path，但文件系统中已不存在

未出现在以上三类中的文件视为**未变化**，跳过后续处理。

> `mtime + size` 的组合可以有效过滤绝大多数未变化的文件，同时避免对每个文件做内容哈希（开销更大）。

### 阶段二：增量 upsert / delete

仅对阶段一识别出的变更文件执行：

- **新增 / 修改**：读取文件内容，解析 frontmatter、wiki-links、embeds、tags，upsert 对应数据库行
- **删除**：从数据库中删除对应行

未变化的文件不读取、不解析、不写库。

### 阶段三：全量 backlinks 重算

无论阶段二处理了多少文件，最后都执行一次全量 backlinks 重算：

```sql
SELECT path, links FROM notes
```

在内存中构建反向索引 `target → [source, ...]`，然后批量写回所有笔记的 `backlinks` 字段。

---

## 3. 为什么 backlinks 必须全量重算

backlinks 是全局反向索引，任何一个笔记的 links 发生变化，都可能影响其他笔记的 backlinks。例如：

- 笔记 A 新增了 `[[C]]`，则 C 的 backlinks 需要加入 A
- 笔记 B 被删除，则所有曾被 B 链接的笔记的 backlinks 都需要移除 B
- rename 操作后，大量笔记的 links 从 `[[old]]` 改为 `[[new]]`

增量计算 backlinks 需要知道"哪些 links 消失了、哪些新增了"，然后做差集运算，逻辑复杂且容易出错。而全量重算只需要一次 `SELECT path, links FROM notes`，数据完全在 DuckDB 内部，不涉及文件 IO，在万级笔记规模下耗时在百毫秒以内，没有性能问题。

---

## 4. 为什么暂不做"只重新索引受影响的文件"的优化

一种更激进的增量方案是：在 rename 或其他操作后，精确追踪"哪些文件受到影响"，只对这些文件重新索引。这个方向暂不推荐，原因如下：

**性能收益不明显。** 增量更新真正节省的开销在于文件 IO 和内容解析，这部分已经通过阶段一的变更检测有效规避。DuckDB 的读写操作本身很快，并非瓶颈。

**复杂度显著上升。** 精确追踪"受影响文件"需要在 rename、delete 等操作中维护额外的元数据，或在 index 时做二次依赖分析，实现复杂，正确性难以保证。

**引入对比并不合算。** 一种变体是"先读现有 backlinks，对比后只写变化行"，但这额外引入了一次全表读取和逐行对比，而原本全量写回本身就很快，对比操作可能得不偿失。

**应在有实测数据后再优化。** 在没有 benchmark 证明存在性能问题之前，引入复杂的增量逻辑是过早优化。正确的顺序是：先上线当前方案，用真实笔记目录测试，发现瓶颈后再针对性优化。

---

## 5. name 唯一性约束与同名文件处理

### name 唯一性假设

markbase 要求笔记目录内所有笔记的 `name`（文件名去除扩展名）全局唯一。这是 wiki-links 语义的基础：`[[note-name]]` 必须能无歧义地指向唯一一个笔记，backlinks 作为 links 的反向索引，存储的也是 name 而非 path，依赖同样的前提。

`note rename` 命令已在写操作层面强制这一约束（同名时拒绝执行）。index 命令需要在读取阶段同样检测并拒绝同名文件。

### index 遇到同名文件的处理

index 扫描过程中，若发现新文件的 name 与已处理文件的 name 冲突：

- **跳过**后发现的冲突文件，不写入数据库
- 向 **stderr** 输出 warning，明确列出两个冲突的完整 path
- 继续处理其余文件，**exit code 仍为 0**（非致命错误）

warning 格式示例：

```
⚠ Skipped: notes/personal/todo.md — name conflict with notes/project/todo.md
```

两个 path 都需要在 warning 中出现，方便用户定位和解决冲突。

### backlinks 重算与 name 的关系

backlinks 重算时，反向索引的 key 直接来自 `links` 字段中存储的 name，与 wiki-links 语义一致，不需要额外的 path → name 转换。name 全局唯一性保证了这个映射是无歧义的。

---

## 6. 与 rename 操作的协作

`note rename` 执行完文件重命名和 wiki-links 替换后，自动触发一次普通的 `index` 调用以更新数据库。

**重要约束：** DuckDB 同一时间只允许一个进程以读写方式打开数据库文件。因此 rename 命令必须在触发 index 之前，显式关闭自身持有的数据库连接，否则 index 无法以读写方式打开数据库。实现上，应确保 `Database` struct 被 drop（连接关闭）后，再执行 index 流程。

index 执行后会：

1. 检测到被重命名的文件（path 变化）和所有被改写 links 的文件（mtime 变化）
2. 对这些文件重新解析并 upsert
3. 全量重算 backlinks，结果自然正确

rename 不需要任何特殊的 backlinks 处理逻辑，复用标准 index 流程即可。

---

## 7. 总结

| 阶段 | 范围 | 理由 |
|---|---|---|
| 变更检测 | 全量扫描文件系统元数据 | 只读 mtime/size，开销极低 |
| upsert / delete | 仅变更文件 | 跳过文件 IO 是增量更新的核心价值 |
| backlinks 重算 | 全量，纯数据库操作 | 正确性优先，性能开销可接受 |

这个方案在正确性、实现复杂度和性能之间取得了合理的平衡，适合作为当前阶段的基准实现。
