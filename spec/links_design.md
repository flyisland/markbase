# Links, Backlinks, and Embeds Design

## 1. Definitions

### 1.1 Links (Outgoing Links)

Links are internal references found in a note. To match Obsidian's behavior, the `links` field stores **both** `[[...]]` wiki-links and `![[...]]` embeds from **two sources**: the note body and the frontmatter properties. Obsidian supports `[[Link]]` syntax inside text-type frontmatter properties (quoted, e.g. `related: "[[other-note]]"`), and these are counted as links just like body references.

Supported syntax variants:

| Syntax | Description |
|---|---|
| `[[note-name]]` | Basic link to a note named `note-name` |
| `[[note-name#Heading]]` | Link to a specific heading section within the target note |
| `[[note-name#^blockid]]` | Link to a specific block (marked with `^blockid`) within the target note |
| `[[note-name\|display text]]` | Custom display text; the actual link target remains `note-name` |
| `[[note-name#Heading\|display text]]` | Link to a section with custom display text |
| `[[folder/note-name]]` | Path-qualified link (generated by Obsidian when duplicate names exist) |

In addition, all `![[...]]` embed references (see §1.3) are also counted as links.

**Storage rule**: The `links` field stores the **target name** extracted from both `[[...]]` and `![[...]]`, including attachment filenames (with extension). Path prefixes are stripped; section anchors (`#...`), block IDs (`#^...`), and display text (`|...`) are discarded.

Examples:
```
[[architecture#Background|Arch Docs]]  →  links: "architecture"
[[notes/design]]                       →  links: "design"
![[diagram.png]]                       →  links: "diagram.png"
![[design#Background]]                 →  links: "design"
```

**Relationship to backlinks**: Because `links` includes all references (both `[[]]` and `![[]]`), the backlinks reverse index also reflects embed relationships. This is consistent with how Obsidian computes backlinks.

### 1.2 Backlinks (Incoming Links)

Backlinks are the **reverse index of links**: for a note B, its `backlinks` field contains the names of all notes whose `links` field includes `B` — whether via `[[B]]` or `![[B]]`.

Backlinks are not extracted directly by `extractor.rs`. Instead, they are fully recomputed in phase 3 of `scanner.rs` (see `index_incremental_design.md` §2): all notes' `path` and `links` are read from the database, a `target → [source, ...]` reverse map is built in memory, and then each note's `backlinks` field is written back in bulk.

### 1.3 Embeds

Embeds are content references written with the `![[...]]` syntax only. An embed means "render the target content inline in the current note." The `embeds` field is a **subset of `links`** — it contains only the targets from `![[...]]` references.

Embeds can reference multiple content types:

| Syntax | Description |
|---|---|
| `![[note-name]]` | Embed the full content of a note |
| `![[note-name#Heading]]` | Embed a specific section of a note |
| `![[note-name#^blockid]]` | Embed a specific block of a note |
| `![[image.png]]` | Embed an image attachment |
| `![[audio.mp3]]` | Embed an audio attachment |
| `![[document.pdf]]` | Embed a PDF attachment |

**Storage rule**: The `embeds` field stores the full target name from `![[...]]` — the portion before `#` or `|` — **including the file extension if present**. Path prefixes are stripped; section anchors and display text are discarded.

Examples:
```
![[architecture#Background]]  →  embeds: "architecture"
![[diagram.png]]               →  embeds: "diagram.png"
![[meeting-notes#^abc123]]     →  embeds: "meeting-notes"
```

**Summary of the relationship between the three fields**:

```
links     = [[...]] from body  ∪  [[...]] from frontmatter  ∪  ![[...]] from body
embeds    = ![[...]] from body only
backlinks = reverse index of links (computed post-indexing)
```

## 2. Extraction Rules (Regex-Based)

Extraction logic lives in `extractor.rs` and operates on a single note, covering both the note body and frontmatter properties.

### 2.1 Embeds First

Embeds and links share nearly identical syntax (embeds just have a `!` prefix). **Embeds must be identified first**, before the general `[[...]]` pattern is applied, so that `![[note]]` is correctly attributed to both `embeds` and `links` rather than being counted only as a plain link.

The shared `EMBED_RE` pattern (defined in §5.1) matches `![[...]]`. The second capture group yields the target name (everything before `#` or `|`). Each match is added to **both** `embeds` and `links`.

### 2.2 Links Extraction

After all `![[...]]` patterns are consumed, the shared `WIKILINK_RE` pattern (defined in §5.1) matches the remaining `[[...]]` for plain wiki-links. The negative lookbehind in the pattern ensures already-processed embed syntax is not re-matched. The second capture group is taken, any path prefix is removed, and the `.md` extension is stripped if present. Each match is added to `links` only (not `embeds`).

### 2.3 Scan Scope

Link extraction runs over **two scopes**:

**Note body**: the full content after the frontmatter has been stripped (frontmatter is pre-extracted by `gray_matter`). Both `[[...]]` and `![[...]]` patterns are matched here.

**Frontmatter properties**: after `gray_matter` parses the YAML, all string-type property values are scanned for `[[...]]` patterns. Only plain wiki-links are expected here (not embeds); any matches are added to `links` but not `embeds`. Note that Obsidian requires internal links in frontmatter to be quoted (e.g. `related: "[[other-note]]"`), so the regex runs on the raw string value after YAML parsing.

Content inside fenced code blocks (` ```...``` `) in the note body must **not** be matched as links, consistent with Obsidian's behavior. This can be achieved by replacing code block content with equal-length placeholder characters before scanning, or by incorporating code-block exclusion into the regex patterns.

## 3. Data Relationship Example

Given three notes:

**`design.md`**
```markdown
---
tags: [architecture]
related: "[[api-spec]]"
---
See [[system-overview]] and [[api-spec#Authentication|Auth Docs]].
![[diagram.png]]
```

**`system-overview.md`**
```markdown
See [[design]] for background.
![[design#Background]]
```

**`api-spec.md`**
```markdown
No links here.
```

After indexing:

| path | links | embeds | backlinks |
|---|---|---|---|
| design.md | [system-overview, api-spec, diagram.png] | [diagram.png] | [system-overview] |
| system-overview.md | [design] | [design] | [design] |
| api-spec.md | [] | [] | [design] |

Notes:
- `design.md`'s `links` includes `api-spec` from both the frontmatter property (`related: "[[api-spec]]"`) and the body link (`[[api-spec#Authentication|Auth Docs]]`); after deduplication only one entry is stored.
- `design.md`'s `links` includes `diagram.png` (from `![[diagram.png]]`). `embeds` contains only `diagram.png`.
- `system-overview.md`'s `links` and `embeds` both include `design` (from `![[design#Background]]`). Because `links` includes it, `design.md`'s `backlinks` reflects `system-overview` as an incoming reference.
- `api-spec.md`'s backlinks includes `design` because `design.md` references `api-spec` via both body and frontmatter.

## 4. Interaction with `note rename`

When `note rename old-name new-name` runs, all wiki-link and embed references across the vault must be updated. This covers two scopes in every file:

**Note body:**
- `[[old-name]]` → `[[new-name]]`
- `[[old-name#Heading]]` → `[[new-name#Heading]]`
- `[[old-name|alias]]` → `[[new-name|alias]]`
- `[[old-name#Heading|alias]]` → `[[new-name#Heading|alias]]`
- `![[old-name]]` → `![[new-name]]`
- `![[old-name#Section]]` → `![[new-name#Section]]`

**Frontmatter properties:**
- `"[[old-name]]"` → `"[[new-name]]"`
- `"[[old-name#Heading|alias]]"` → `"[[new-name#Heading|alias]]"`
- (and all other `[[...]]` variants, quoted as required by Obsidian)

The same `WIKILINK_RE` pattern is reused for frontmatter scanning. Because the link syntax inside a YAML string value is identical to the body syntax, no separate regex is needed — only the scope of text to scan differs.

**Which files to scan: backlinks lookup vs. full vault scan**

Two strategies are possible for deciding which files to rewrite:

- **Backlinks lookup**: query the database for `backlinks` of `old-name` to get the exact set of files that reference it, then rewrite only those files.
- **Full vault scan**: walk every file in the vault and apply the regex unconditionally.

The backlinks lookup is appealing because it avoids reading files that have no references to the renamed note. However, it has a correctness risk: **the database index may be stale**. If a file was modified outside of `markbase index` (e.g. edited directly in Obsidian while the index was not running), its links may not be reflected in `backlinks` yet. Relying on backlinks alone would silently miss those files.

Full vault scan is therefore the safer default. The performance cost is bounded by file I/O across the vault, which for typical note collections (thousands of files, mostly small text files) completes in well under a second. Correctness is not dependent on index freshness.

The recommended approach is **full vault scan**. If benchmarking on a very large vault reveals a measurable performance problem, backlinks lookup can be introduced as an optimization at that point — but only after the correctness tradeoff is explicitly accepted.

Once all files are rewritten, `renamer.rs` triggers `markbase index` (after explicitly closing its own database connection first — see `index_incremental_design.md` §6). The standard three-phase index flow then rebuilds `links`, `embeds`, and `backlinks` from scratch.

## 5. Implementation Requirements

### 5.1 Shared Regex Patterns and Extraction Functions

All wiki-link and embed recognition must go through a **single shared parsing module** to guarantee consistent behavior across the codebase. The same regex logic must not be duplicated in different places.

The root of the shared logic is the **regex patterns** themselves. Both extraction (used during indexing) and rewriting (used during rename) operate on the same syntax, so both must derive from the same pattern definitions. `extractor.rs` should own these patterns and expose them as public constants or lazy statics, alongside the higher-level extraction functions:

```rust
// Shared pattern constants — the single source of truth for wikilink syntax
pub static EMBED_RE: Lazy<Regex> = Lazy::new(||
    Regex::new(r"(!\[\[)([^\[\]|#]+?)((?:[#|][^\[\]]*)?)\]\]").unwrap()
);
pub static WIKILINK_RE: Lazy<Regex> = Lazy::new(||
    Regex::new(r"((?<!!))\[\[([^\[\]|#]+?)((?:[#|][^\[\]]*)?)\]\]").unwrap()
);

/// Extracts all embed targets from the note body.
/// Each result is added to both `links` and `embeds`.
pub fn extract_embeds(body: &str) -> Vec<String> { ... }

/// Extracts all plain wiki-link targets from the note body (excluding embeds).
/// Each result is added to `links` only.
pub fn extract_wikilinks(body: &str) -> Vec<String> { ... }

/// Extracts all wiki-link targets from frontmatter string-type property values.
/// Each result is added to `links` only (not `embeds`).
pub fn extract_frontmatter_links(frontmatter: &Matter) -> Vec<String> { ... }
```

`scanner.rs` calls `extract_embeds`, `extract_wikilinks`, and `extract_frontmatter_links` during indexing to build the complete `links` and `embeds` fields. `renamer.rs` uses `EMBED_RE` and `WIKILINK_RE` directly to perform in-place substitution on both body and frontmatter. Neither module defines its own patterns — the regex lives in exactly one place.

### 5.2 Rename Rewrite Implementation

The rename rewrite in `renamer.rs` must replace the note name in both `[[...]]` and `![[...]]` contexts while preserving all other parts of the link (section anchors, block IDs, display text, path prefixes). This applies to **both the note body and frontmatter property values**.

**Recommended approach**: reuse `EMBED_RE` and `WIKILINK_RE` from `extractor.rs` across three sequential passes — embeds in the body, plain wiki-links in the body, and wiki-links in frontmatter property values.

Before running any regex, perform a cheap plain-text check to skip files that cannot possibly contain a reference to `old-name`. Only files that pass this check are read fully, processed, and written back:

```rust
// Fast pre-check — skip files that cannot contain a reference to old-name.
// Reading the raw file bytes and doing a substring search is far cheaper than
// running the full regex on every file in the vault.
let raw = fs::read_to_string(&file_path)?;
if !raw.contains(old_name) {
    continue;
}

// Pass 1: rewrite embeds in body — ![[old-name...]] → ![[new-name...]]
let body = EMBED_RE.replace_all(&body, |caps: &Captures| {
    if &caps[2] == old_name {
        format!("{}{}{}]]", &caps[1], new_name, &caps[3])
    } else {
        caps[0].to_string()
    }
});

// Pass 2: rewrite plain wiki-links in body — [[old-name...]] → [[new-name...]]
let body = WIKILINK_RE.replace_all(&body, |caps: &Captures| {
    if &caps[2] == old_name {
        format!("[[{}{}]]", new_name, &caps[3])
    } else {
        caps[0].to_string()
    }
});

// Pass 3: rewrite wiki-links in frontmatter property values
// Iterate over all string-type property values parsed by gray_matter,
// apply WIKILINK_RE replacement on each value, then serialize back to YAML.
for value in frontmatter.string_values_mut() {
    *value = WIKILINK_RE.replace_all(value, |caps: &Captures| {
        if &caps[2] == old_name {
            format!("[[{}{}]]", new_name, &caps[3])
        } else {
            caps[0].to_string()
        }
    }).to_string();
}
```

The capture groups mirror the shared pattern structure: group 1 is the opening bracket sequence (including `!` for embeds), group 2 is the note name, group 3 is the remainder (anchor, display text). Only group 2 is replaced; everything else is preserved verbatim.

This approach correctly handles all variants:

```
# Body
[[old-name]]               →  [[new-name]]
[[old-name#Heading]]       →  [[new-name#Heading]]
[[old-name|alias]]         →  [[new-name|alias]]
[[old-name#Heading|alias]] →  [[new-name#Heading|alias]]
![[old-name]]              →  ![[new-name]]
![[old-name#Section]]      →  ![[new-name#Section]]

# Frontmatter property value (the surrounding quotes are part of YAML, not the link)
"[[old-name]]"             →  "[[new-name]]"
"[[old-name|alias]]"       →  "[[new-name|alias]]"
```

```

After all three passes, compare the result against the original content before writing. If nothing changed — for example, `old-name` appeared in the file as plain text but not inside any `[[...]]` — skip the write entirely:

```rust
if new_content != raw {
    fs::write(&file_path, &new_content)?;
}
```

**Boundary condition**: the regex matches `old-name` as a complete name segment, not as a substring. For example, renaming `foo` does not affect `[[foobar]]`, because the name group is always terminated by `#`, `|`, or `]]`.
